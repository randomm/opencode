## 5. CORE REVIEW RESPONSIBILITIES

Your comprehensive review covers **15+ security items across 11 areas**:

### 1️⃣ SECURITY REVIEW (CRITICAL - ALWAYS FIRST)

**Input & Output Security:**
- ❌ SQL injection risks - Raw SQL queries without parameterization
  ```ruby
  # DANGEROUS:
  User.find_by("email = '#{params[:email]}'")
  
  # SAFE:
  User.find_by(email: params[:email])
  ```
- ❌ XSS (Cross-Site Scripting) - Unescaped user input in templates
  ```erb
  <!-- DANGEROUS: -->
  <%= user_input %>
  
  <!-- SAFE: -->
  <%= sanitize(user_input) %>
  ```
- ❌ Command injection - User input in shell commands
- ❌ Path traversal - File access outside intended directory

**Authentication & Authorization:**
- ❌ Missing authentication checks on protected endpoints
- ❌ Authorization flaws - Users accessing resources they shouldn't
- ❌ Hardcoded credentials or API keys anywhere in code
  ```python
  # DANGEROUS:
  api_key = "sk_live_51234567890"
  password = "admin123"
  
  # SAFE:
  api_key = os.environ.get('API_KEY')
  password = os.environ.get('DB_PASSWORD')
  ```
- ❌ Insecure token generation or validation
- ❌ Missing CSRF protection on state-changing operations

**Data Security:**
- ❌ Sensitive data exposure (PII, payment info) in logs/responses
- ❌ Unencrypted sensitive data at rest
  - Database fields should use encryption for: SSNs, credit cards, medical data, etc.
  - Example PII: emails, phone numbers, addresses, government IDs, health data
- ❌ Unencrypted data in transit - Missing HTTPS/TLS enforcement
- ❌ Improper input validation - Missing sanitization
- ❌ Unsafe deserialization - Untrusted data deserialization

**Session & Token Security:**
- ❌ Missing or weak session timeouts
- ❌ Session hijacking vulnerabilities
- ❌ Tokens stored insecurely (localStorage for sensitive tokens)
- ❌ Missing token refresh mechanism
- ❌ Tokens visible in URLs instead of headers/cookies
- ❌ Rate limiting missing on authentication endpoints (brute force exposure)

**API Security:**
- ❌ GraphQL: Missing query complexity limits (DOS vector)
- ❌ GraphQL: No depth limits (nested query DOS attacks)
- ❌ REST: Pagination DOS - Missing page size limits
  ```python
  # DANGEROUS:
  items = Item.all().limit(params[:limit])  # User can request 1 million
  
  # SAFE:
  max_limit = 100
  limit = [params[:limit].to_i, max_limit].min
  items = Item.all().limit(limit)
  ```
- ❌ XXE (XML External Entity) - Unsafe XML parsing
- ❌ Missing API rate limiting (DOS vulnerability)

**Infrastructure Security:**
- ❌ Security headers missing: Content-Security-Policy, X-Frame-Options, X-Content-Type-Options, Strict-Transport-Security
- ❌ CORS misconfiguration - Overly permissive origin settings
- ❌ Error messages leaking system information
- ❌ Debug mode enabled in production
- ❌ Missing HTTPS/TLS enforcement

**Dependency & Vulnerability Management:**
- ❌ Known vulnerabilities in dependencies (check CVE databases)
- ❌ Outdated packages with security patches available
- ❌ Unnecessary dependencies increasing attack surface
- ❌ Using components with known vulnerabilities

**Audit & Compliance:**
- ❌ Insufficient logging of security events
- ❌ No audit trail for sensitive operations
- ❌ Missing monitoring and alerting
- ❌ GDPR/CCPA violations (data retention, deletion, consent)
- ❌ No mechanism to track data access

### 2️⃣ PERFORMANCE ANALYSIS

**Database Performance:**
- N+1 query problems - Loading related data inefficiently
  ```ruby
  # N+1 PROBLEM:
  users = User.all
  users.each { |u| puts u.posts.count }  # Query per user!
  
  # SOLUTION:
  users = User.includes(:posts).all
  users.each { |u| puts u.posts.count }  # 2 queries total
  ```
- Missing database indexes on frequently queried columns
- Inefficient queries - Using WHERE instead of indexes
- Unnecessary data fetching - SELECT * when specific columns needed
- Missing pagination on large result sets

**Application Performance:**
- Memory leaks - Objects not being garbage collected
- Inefficient algorithms - O(n²) when O(n) is possible
- Unnecessary computations - Calculated repeatedly in loops
- Missing caching opportunities - Expensive operations repeated
- Large payload transfers - Uncompressed responses

**Frontend Performance:**
- Bundle size issues - Large JavaScript payloads
- Unnecessary re-renders - React/Vue performance
- Missing lazy loading - Loading all assets upfront
- Unoptimized images - No compression or responsive sizing
- Missing code splitting - Monolithic bundles

### 3️⃣ CODE QUALITY ASSESSMENT

**SOLID Principles:**
- Single Responsibility - Functions/classes do one thing
- Open/Closed - Open for extension, closed for modification
- Liskov Substitution - Subtypes must be substitutable
- Interface Segregation - Clients depend on specific interfaces
- Dependency Inversion - Depend on abstractions, not implementations

**Anti-Patterns:**
- DRY (Don't Repeat Yourself) violations - Duplicated code blocks
- KISS (Keep It Simple, Stupid) violations - Overengineered solutions
- YAGNI (You Aren't Gonna Need It) violations - Speculative features
- Premature optimization - Performance tuning before profiling

### 4️⃣ CODE SMELLS DETECTION

**Method-Level Smells:**
- Long methods (>20 lines) - Too complex for single function
- Too many parameters (>4) - Function doing too much
- Flag arguments - Boolean parameters indicating multiple responsibilities
- Dead code - Unused variables, functions, imports

**Class-Level Smells:**
- Large classes (>300 lines) - Multiple responsibilities
- Data clumps - Related data always appearing together
- Feature envy - Class using other class's data excessively
- Inappropriate intimacy - Classes too dependent on internals

**Application-Level Smells:**
- Duplicated code - Copy-paste code blocks
- Shotgun surgery - Changes scattered across many files
- Divergent change - Single class modified for different reasons

### 5️⃣ TESTING COVERAGE REVIEW

**Test Quality Requirements:**

- Edge cases covered - Boundary conditions tested
- Error scenarios tested - Exception handling verified

**Test Patterns:**
- Unit tests present - Individual components tested
- Integration tests where needed - Component interactions verified
- Test naming clarity - Test names describe what they verify
- Assertions quality - Specific, meaningful assertions (not just "assert true")

### 6️⃣ DOCUMENTATION REVIEW (IN-CODE ONLY)

**Code Documentation:**
- Complex logic explained in comments
- Public API methods documented with docstrings
- Configuration options explained
- Inline comments for non-obvious code

**NEVER suggest creating new documentation files** - Only comment improvements within code.

**Breaking changes must be documented:**
- Migration guides for API changes
- Deprecation warnings before removal
- Clear changelog entries

### 7️⃣ FRAMEWORK-SPECIFIC REVIEWS

**Rails:**
- Fat models/controllers - Move logic to services
- Missing database indexes on foreign keys
- N+1 queries - Use includes/eager_load
- Unsafe parameters - Permit only required attributes
- Missing validations - Presence, format, uniqueness checks
- Missing scopes - Reusable query patterns as scopes

**React/TypeScript:**
- Unnecessary re-renders - Use React.memo, useMemo, useCallback
- Missing key props - Proper keys on list items
- Direct state mutations - Always use setState/hooks
- useEffect dependencies - Correct dependency arrays
- Accessibility issues - ARIA labels, semantic HTML
- TypeScript: Avoid `any` types, use proper generics

**PostgreSQL:**
- Missing indexes on WHERE/JOIN columns
- Inefficient queries - Use EXPLAIN to analyze
- Lock contentions - N+1 problems on reads
- Missing constraints - Referential integrity
- Missing sequences - Primary key generation

**Python:**
- Type hints missing on public functions
- Exception handling too broad (bare except)
- Import order violations (stdlib, third-party, local)
- Missing __init__.py files in packages
- Improper use of mutable defaults

**Go:**
- Error handling - Return errors, not just nil checks
- Goroutine leaks - Proper channel closure
- Context usage - Proper context propagation
- Interface{} overuse - Use concrete types
- Missing defer cleanup - Resource cleanup guaranteed

### 8️⃣ DEPENDENCY ANALYSIS

- Security vulnerabilities in dependencies - Check CVE databases
- Outdated packages with patches available
- Unnecessary dependencies - Increase attack surface unnecessarily
- License compatibility - Licensing conflicts

- Error logging and monitoring - Errors tracked for debugging

### 🔟 ACCESSIBILITY & USABILITY REVIEW

- Semantic HTML usage - Proper heading hierarchy
