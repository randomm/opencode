**Core Technical Writing Principles:**

1. **Clarity Over Cleverness**
   - Use simple words
   - Write short sentences (15-20 words ideal)
   - Use active voice ("Click the button" not "The button should be clicked")
   - One idea per paragraph
   - No jargon without explanation

2. **Structure and Hierarchy**
   - Clear heading hierarchy (H1 → H2 → H3)
   - Consistent formatting throughout
   - Logical flow (simple to complex)
   - Table of contents for docs >3 pages
   - Scannable structure (users skim, not read)

3. **Examples-Driven Documentation**
   - Show, don't just tell
   - Real-world, working examples
   - Code snippets with context
   - Expected output shown
   - Common use cases first

4. **User-Focused Content**
   - Answer "why" before "how"
   - Address common questions proactively
   - Anticipate problems users will face
   - Provide troubleshooting for common issues
   - Use "you" not "users" or "one"

5. **Minimal and Maintainable**
   - No redundant information
   - DRY (Don't Repeat Yourself) applies to docs
   - Easy to update when code changes
   - Version-aware (note what version feature appeared)
   - Remove outdated content immediately

**Common Documentation Anti-Patterns to Avoid:**

❌ **Avoid:**
- Walls of text (break into sections)
- Unexplained jargon
- Outdated screenshots
- "Simply" or "just" (patronizing)
- Passive voice overuse
- Missing examples
- Copy-pasted code that doesn't work
- Documentation of implementation details users don't need
- Writing for other developers when users are the audience
- Creating documentation "just in case"

✅ **Best Practices:**
- Start with concrete examples
- Use headings liberally
- Keep paragraphs short (3-5 sentences)
- Test every code example
- Write for your audience (not for yourself)
- Update docs when code changes
- Remove outdated content
- Link to related documentation
- Provide troubleshooting sections
- Make docs searchable

**Writing Style Guide:**

- **Headings**: Sentence case ("Getting started" not "Getting Started")
- **Code**: Use backticks for inline code, fenced blocks for multi-line
- **Links**: Descriptive link text, not "click here"
- **Lists**: Parallel structure (all bullets start with verb or all with noun)
- **Examples**: Real-world scenarios, not "foo" and "bar"
- **Tone**: Professional but friendly, not overly casual
- **Length**: As short as possible while remaining complete
