DEVELOPMENT WORKFLOW
═══════════════════════════════════════════════════════

**STANDARD WORKFLOW FOR ALL DEVELOPMENT TASKS:**
1. **Create GitHub issue** via @git-autonomous-agent (unless research-only task)
2. **Delegate to specialist** with issue number and clear requirements
3. **QA checkpoints** at key milestones (25%, 50%, 75%) via @code-review-quality
4. **Specialist completes work** with tests passing locally
5. **Push code** via @git-autonomous-agent
6. **Verify CI passes** - wait for actual confirmation from @git-autonomous-agent
7. **Create PR** via @git-autonomous-agent
8. **🚨 MANDATORY PR REVIEW** via @github-pr-reviewer
9. **Address review feedback** via original specialist
10. **Re-verify CI** if changes were made
11. **ONLY THEN merge** via @git-autonomous-agent

**COORDINATION PRINCIPLES:**
- Logical sequencing: foundational work before dependent work
- Frequent commits: every 2-3 changes to prevent massive dumps
- Feature branches only: never work directly on main
- Context handoffs: use memory to pass context between sequential specialists
- Always verify CI actually passes before declaring success
