**CORE REACT DEVELOPMENT PRINCIPLES:**

1. **Component Architecture**
   - Design with composition over inheritance
   - Create small, focused, reusable components
   - Follow single responsibility principle
   - Use proper component naming (PascalCase)
   - Organize components by feature/domain
   - Implement proper prop drilling prevention (Context, composition)
   - Use compound component patterns when appropriate

2. **TypeScript Excellence**
   - Use strict TypeScript configuration
   - Define comprehensive interfaces for all props
   - Avoid using 'any' type - use unknown or generics
   - Create reusable type utilities
   - Use discriminated unions for complex state
   - Implement proper type guards
   - Export types alongside components
   - Let TypeScript infer when obvious

3. **State Management Best Practices**
   - Use appropriate solution for scale:
     * useState for local component state
     * useContext for cross-component state (theme, auth)
     * useReducer for complex state logic
     * Zustand/Redux Toolkit for app-wide state
   - Keep state as local as possible
   - Normalize complex state structures
   - Store patterns in memory for future decisions

4. **React Hooks Mastery**
   - Follow Rules of Hooks strictly
   - Create custom hooks for reusable logic
   - Use proper hook dependencies
   - Implement cleanup in useEffect
   - Optimize with useMemo and useCallback appropriately
   - Avoid unnecessary re-renders
   - Document custom hooks thoroughly

**MINIMALIST PHILOSOPHY & LESS IS MORE:**

1. **Zero Bloat Tolerance**
   - Build ABSOLUTE MINIMUM to meet requirements
   - NO extra files, NO premature abstractions, NO "nice-to-haves"
   - Every line must have clear purpose tied to failing test
   - Question EVERY dependency - prefer built-in solutions
   - fetch() not axios (unless specific features needed)
   - Web Components over framework components when reasonable
   - CSS not CSS-in-JS when possible

2. **Framework Decision Tree**
   ```
   Can vanilla JS do this? → Research with Perplexity → Use vanilla
   Need state management? → Check memory for patterns → Try URL params/localStorage
   Need routing? → Research native alternatives → Use History API first
   Need component? → Check if Web Component sufficient → Use React only if needed
   Still need more? → Research minimal tools → Choose smallest that works
   ```

3. **Dependency Decision Tree**
   ```
   Can native APIs do this? → Perplexity check → Use native
   Is it <20 lines to implement? → Check memory → Write those lines
   Will we use it >3 times? → Still consider writing it
   Is it genuinely complex? → Research alternatives → MAYBE add dependency
   ```

4. **Vanilla/Native API Approach**
   **ALWAYS check with Perplexity for native alternatives first**
   - Research: "Native alternative to [library]"
   - Research: "Can this UI be built with vanilla JS?"
   - Research: "Web Component vs React component for [use case]"
   - Document decisions: "Used native FormData instead of form library"
   - Store vanilla patterns that replaced frameworks in memory

**TESTING STANDARDS:**

**MANDATORY before delegating to @git-agent:**
- Achieve minimum 80% test coverage (95%+ for critical paths like auth, payments)
- Write the SIMPLEST failing test that captures the requirement
- Implement ONLY enough code to make it pass
- Test behavior, not implementation

**Testing Framework Decision - Jest vs Vitest:**
- Jest v29.7: 44k stars, excellent ecosystem, comprehensive docs, 8s startup time, Istanbul coverage
- Vitest v1.3.1: 500ms startup (16x faster), native ESM support, 95% Jest API compatibility, v8 coverage provider
- **BOTH valid - Choose based on project constraints:**
  * Jest: Mature ecosystem, team familiarity, CRA projects, established patterns
  * Vitest: Speed critical, modern build tooling, TypeScript-first, faster CI feedback loops
- **Store decision rationale in memory for future projects**

**Test Types to Implement:**
- Unit tests: Pure functions, utilities, hooks
- Component tests with React Testing Library: User behavior, not implementation
- Integration tests: Feature flows, multiple components
- Accessibility tests with jest-axe: WCAG compliance
- Snapshot tests: Stable components only (not overused)

**Testing Best Practices:**
```javascript
// YES: Test behavior users care about
test('adds item to cart', () => {
  expect(calculateTotal([10, 20])).toBe(30)
})

// NO: Test implementation details
test('calls reduce with sum function', () => {...})
```

- One assertion per test when possible
- Use proper queries (getByRole > getByText > getByTestId)
- Mock only external dependencies
- Test error states and edge cases
