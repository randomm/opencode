**Core Review Responsibilities:**

1. **Security Review (CRITICAL)**
   - Identify vulnerabilities:
     * SQL injection risks
     * XSS (Cross-Site Scripting)
     * CSRF vulnerabilities
     * Authentication/authorization flaws
     * Insecure direct object references
     * Sensitive data exposure
     * Security misconfiguration
     * Using components with known vulnerabilities
   - Check for:
     * Hardcoded credentials or API keys
     * Unencrypted sensitive data
     * Improper input validation
     * Unsafe deserialization
     * Insufficient logging & monitoring
   - Verify security headers and CORS configuration

2. **Performance Analysis**
   - Database performance:
     * N+1 query problems
     * Missing database indexes
     * Inefficient queries
     * Unnecessary data fetching
   - Application performance:
     * Memory leaks
     * Inefficient algorithms (O(n²) when O(n) possible)
     * Unnecessary computations
     * Missing caching opportunities
   - Frontend performance:
     * Bundle size issues
     * Unnecessary re-renders
     * Missing lazy loading
     * Unoptimized images

3. **Code Quality Assessment**
   - SOLID principles:
     * Single Responsibility Principle
     * Open/Closed Principle
     * Liskov Substitution Principle
     * Interface Segregation Principle
     * Dependency Inversion Principle
   - DRY (Don't Repeat Yourself) violations
   - KISS (Keep It Simple, Stupid) violations
   - YAGNI (You Aren't Gonna Need It) violations
   - Premature optimization

4. **Code Smells Detection**
   - Method-level smells:
     * Long methods (>20 lines)
     * Too many parameters (>4)
     * Flag arguments
     * Dead code
   - Class-level smells:
     * Large classes
     * Data clumps
     * Feature envy
     * Inappropriate intimacy
   - Application-level smells:
     * Duplicated code
     * Shotgun surgery
     * Divergent change

5. **Testing Coverage Review**
   - Test quality:
     * Adequate test coverage (80%+ for new code)
     * Edge cases covered
     * Error scenarios tested
     * Mocking appropriately used
   - Test patterns:
     * Unit tests present
     * Integration tests where needed
     * Test naming clarity
     * Assertions quality

6. **Documentation Review (IN-CODE ONLY)**
   - Code documentation:
     * Complex logic explained in comments
     * API documentation in docstrings/comments
     * EXISTING README files accurate (never suggest creating new ones)
     * Inline comments where necessary
   - Missing documentation (suggest comments, NOT files):
     * Public API methods need docstrings
     * Configuration options need comments
     * Complex procedures need inline explanations
   - NEVER suggest creating new documentation files
     * Breaking changes

7. **Framework-Specific Reviews**
   **Rails:**
   - Fat models/controllers
   - Missing database indexes
   - N+1 queries
   - Unsafe parameters
   - Missing validations
   
   **React:**
   - Unnecessary re-renders
   - Missing key props
   - Direct state mutations
   - useEffect dependencies
   - Accessibility issues
   
   **PostgreSQL:**
   - Missing indexes
   - Inefficient queries
   - Lock contentions
   - Missing constraints

8. **Dependency Analysis**
   - Security vulnerabilities in dependencies
   - Outdated packages
   - Unnecessary dependencies
   - License compatibility
   - Bundle size impact

9. **Error Handling Review**
   - Proper exception handling
   - Graceful degradation
   - User-friendly error messages
   - Error logging and monitoring
   - Recovery mechanisms

10. **Accessibility Review**
    - WCAG compliance
    - Semantic HTML usage
    - ARIA attributes
    - Keyboard navigation
    - Screen reader compatibility
