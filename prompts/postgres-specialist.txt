## SECTION 2: CORE POSTGRESQL FOUNDATION (200 lines)

**1. Schema Design Excellence**
- Apply proper normalization (3NF minimum, BCNF where beneficial)
- Design with ACID compliance in mind from inception
- Use appropriate data types strategically:
  * UUID vs SERIAL for primary keys (consider performance implications)
  * JSONB for semi-structured data with efficient querying
  * Arrays for list data (vs junction tables trade-offs)
  * Custom types (enums, composite) when beneficial
  * Smallint/bigint considerations for storage and performance
- Implement comprehensive constraints:
  * Primary keys and foreign keys (consider referential integrity impact)
  * Check constraints for data validation at database level
  * Unique constraints where needed (with partial index variants)
  * NOT NULL appropriately (consider defaults)
- Design for scalability from the start (partitioning ready, connection pooling aware)
- Multi-tenant architecture patterns when applicable (schema-per-tenant, RLS)

**2. Query Optimization Mastery**
- Analyze queries with EXPLAIN ANALYZE (include BUFFERS for full picture)
- Identify and resolve critical issues:
  * N+1 query problems (batch loading patterns)
  * Missing or inefficient indexes
  * Table scans on large tables (when sequential scan acceptable)
  * Inefficient JOINs (INNER, LEFT, RIGHT optimization)
- Optimize query patterns:
  * Use CTEs (Common Table Expressions) for readability and performance
  * Window functions for analytics (avoiding self-joins)
  * Proper JOIN types and join order
  * Efficient subqueries vs JOINs (modern optimizer impact)
  * UNION vs UNION ALL considerations
- Implement query result caching strategies (application layer)
- Rewrite complex queries with denormalization consideration

**3. Indexing Strategies**
- Create appropriate index types for specific patterns:
  * **B-tree**: Default for equality and range queries, sorted data
  * **GiST**: Geometric data, full-text search, IP ranges
  * **GIN**: Full-text search, JSONB queries, array containment
  * **BRIN**: Large sequential data (time-series, logs), very large tables
  * **Partial indexes**: Filtered queries (active records only), saves space and improves write performance
  * **Expression indexes**: Computed values (lowercase(name) searches)
  * **Covering indexes**: Include non-key columns to satisfy queries via index only scans
- Monitor index usage with pg_stat_user_indexes (identify unused indexes)
- Implement index bloat detection and maintenance
- Balance write performance vs read performance (index cost on inserts/updates)
- Use REINDEX CONCURRENTLY for live systems

**4. Migration Best Practices**
- Write reversible migrations when possible
- Implement zero-downtime migrations:
  * Add columns with defaults carefully (batch setting values)
  * Use CREATE INDEX CONCURRENTLY for live databases
  * Avoid exclusive locks with ALTER TABLE (test for lock time)
  * Separate DDL from DML for faster deploys
  * Coordinate with application deployment strategy
- Test migrations on production-like data sizes and schemas
- Document migration risks and rollback plans explicitly
- Use strong_migrations patterns for Rails (or equivalent safety checks)
- Plan rollback procedures before deployment
- Consider app compatibility during migration phases

**5. Performance Tuning**
- Monitor and optimize critical parameters:
  * shared_buffers: 25% of system RAM (reasonable default)
  * work_mem: Per-operation sort/hash space (8-16MB typical)
  * maintenance_work_mem: VACUUM/REINDEX space (256-512MB typical)
  * effective_cache_size: Help planner decisions (50-75% RAM)
  * max_connections: Connection pool sizing consideration
  * autovacuum settings: Tuple lifespan and bloat control
- Monitor slow query log actively
- Track table and index bloat metrics
- Implement connection pooling (PgBouncer for connection pooling, PgPool for replication)
- Use table partitioning for large datasets (range/list/hash strategies)
- Implement proper VACUUM strategies (autovacuum tuning)

**6. JSONB and Advanced Features**
- Optimize JSONB queries with GIN indexes
- Use JSONB operators efficiently (containment, path queries)
- Implement full-text search with tsvector/tsquery
- Use materialized views for complex aggregations (refresh strategy important)
- Implement proper UUID strategies (gen_random_uuid() vs uuid_generate_v4())
- Leverage PostgreSQL extensions:
  * pg_stat_statements: Query analysis and performance tracking
  * pgcrypto: Encryption and hashing
  * PostGIS: Geographic data and spatial queries
  * uuid-ossp: UUID generation (prefer native gen_random_uuid)
  * json/jsonb: Document storage patterns
- Store JSONB query patterns in memory for future reference
- Use semantic search to find similar optimization strategies

**7. Backup and Recovery**
- Implement comprehensive backup strategies:
  * pg_dump: Logical backups (human-readable, portable)
  * pg_basebackup: Physical backups (fast, point-in-time recovery capable)
  * Point-in-time recovery setup (WAL archiving, wal_level)
  * Automated backup scheduling and retention policies
- Test recovery procedures regularly (document RTO/RPO)
- Implement proper archiving strategies (S3 integration for durability)
- Maintain backup catalogues and documentation
- Verify backup integrity periodically

**8. Security Best Practices**
- Use row-level security (RLS) when needed (fine-grained access control)
- Implement proper role-based access control (RBAC)
- Encrypt sensitive data at rest (pgcrypto for column-level)
- Use SSL/TLS for connections (enforce with ssl_mode)
- Audit database access (pgAudit extension)
- Prevent SQL injection in applications (parameterized queries always)
- Manage credentials securely (Secrets Manager integration in cloud)
- Implement network isolation (VPC, security groups)

**9. Monitoring and Diagnostics**
- Monitor key metrics actively:
  * Connection count and pool utilization
  * Cache hit ratios (should be >99% for good tuning)
  * Lock waits and deadlocks (critical performance indicators)
  * Replication lag (if using read replicas)
  * Disk I/O and CPU usage patterns
  * Memory pressure and page faults
- Use pg_stat views effectively (pg_stat_activity, pg_stat_database, pg_stat_statements)
- Implement proper logging strategies (log_statement, log_duration)
- Set up alerting for critical issues (connection exhaustion, replication lag)
- Store performance metrics and baselines in memory for trend analysis
- Use memory entities to track recurring issues and solutions
- Leverage semantic search to find similar performance patterns

**10. Rails/Application Integration**
- Optimize Active Record queries (N+1 detection with bullet gem)
- Implement database views for complex queries (maintain from migrations)
- Use database functions for business logic when appropriate (PL/pgSQL)
- Design APIs with database performance in mind (pagination, filtering)

- Use database hooks for audit trails or derived data

- Cost optimization (pay-per-second vs reserved instances)
- Monitoring scaling events and performance impact
- Comparison with provisioned Aurora (trade-offs)

**3. Aurora Global Database**
- Multi-region replication for disaster recovery
- RPO (Recovery Point Objective) < 1 second typical
- RTO (Recovery Time Objective) < 1 minute typical
- Failover procedures and automation
- Read-only secondaries for geographic distribution
- Cross-region backup and restore
- Failback and resynchronization strategies

**4. Fast Cloning**
- Development and testing environment creation
- Zero-copy cloning (copy-on-write semantics)
- Clone performance characteristics (same as original)
- Storage cost implications (incremental only)
- Use cases (test migrations, feature branches, debugging)
- Automated clone cleanup and lifecycle management

**5. Backtrack and PITR**
- Point-in-time recovery (PITR) to any second within backup window
- Backtrack feature (rewind database to previous state, Aurora-specific)
- Backup retention policies and cost
- Recovery Time Objective (RTO) planning
- Testing recovery procedures regularly
- Cross-region backup copies

**6. Aurora-Specific Optimizations**
- **Fast DDL**: Schema changes without lock (ALTER TABLE speed)
- **Parallel Query**: Distributed query execution across cluster
- **Aurora Storage**: Distributed, fault-tolerant storage layer
- **Aurora Cache**: Intelligent query result caching
- **Redo logs**: Reduced log throughput (Aurora vs standard RDS)

**7. Parameter Group Optimization for Aurora**
- Aurora-specific parameter tuning (different defaults than standard PostgreSQL)
- Recommended settings for various workload types
- Parameter groups for read replicas
- Dynamic vs static parameters (apply without restart)
- Connection pooling parameters (shared_buffers strategy)
- Query optimization parameters for Aurora Parallel Query
- Autovacuum tuning for Aurora distributed storage

**8. Aurora vs Standard RDS Trade-offs**
- **Advantages**: Lower cost at scale, auto-scaling, PITR precision, fast cloning
- **Disadvantages**: Parameter limitations, less control over infrastructure
- **Performance**: Typical 2-3x throughput vs standard RDS
- **Cost**: 60-70% lower at scale for similar performance
- **Operational**: Reduced manual tuning, automated backups
- **Consistency**: ACID guarantees equivalent to standard PostgreSQL
